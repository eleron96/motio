# Frontend Boundaries

Документ фиксирует минимальные архитектурные границы для `src/`, чтобы код оставался расширяемым и предсказуемым.

## Слои

1. `features/*/pages`
- Точка сборки экрана.
- Можно связывать UI, store-actions и feature hooks.
- Нельзя хранить сложные доменные преобразования прямо в компоненте страницы.

2. `features/*/components`
- Изолированные UI-блоки feature.
- Локальное состояние и пользовательские обработчики допустимы.
- Доменная логика выносится в `shared/domain` или `features/*/lib`.

3. `features/*/store`
- Состояние и мутации уровня фичи.
- Работа с API, optimistic updates, reconcile-логика.
- Переиспользуемые преобразования данных не дублируются: только через shared helpers.

4. `shared/domain`
- Чистые функции преобразования и бизнес-правила без UI-зависимостей.
- Обязательны unit-тесты на граничные кейсы.

5. `shared/lib`
- Инфраструктурные утилиты (форматирование, network-guards, вспомогательные сервисы).
- Можно зависеть от внешних библиотек, но API должен быть узким и стабильным.

6. `shared/ui`
- Переиспользуемые презентационные компоненты без предметной логики фич.

## Правила зависимостей

1. `pages` -> можно импортировать из `features/*`, `shared/*`.
2. `components` -> можно импортировать из `features/*/hooks|lib` и `shared/*`.
3. `shared/domain` -> не импортирует `features/*/pages` или `shared/ui`.
4. Любая повторяющаяся логика в двух и более местах переносится в `shared/domain` или `shared/lib`.

## Расширение без регрессий

1. Добавить поведение как чистую функцию в `shared/domain` (если это правило/маппинг).
2. Добавить unit-тест на новый helper.
3. Подключить helper в `store`/`page`, не копируя алгоритм.
4. Если затрагивается пользовательский сценарий, добавить спецификацию в `docs/specifications/`.
